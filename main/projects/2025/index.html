    Leveraging Open Source AI in Democratizing Access to Verifiable Credentials and Decentralized Identifiers¶

    2025-LF-Decentralized-Trust Coding TAC Review

    This project focuses on democratizing access to Verifiable Credentials (VCs) and Decentralized Identifiers (DIDs) by integrating open-source AI technologies. Despite the potential of decentralized identity, existing solutions are hindered by complexity, interoperability challenges, and limited user-friendliness. Our objective is to develop an open-source framework that utilizes AI models, particularly large or small language models (LLM/SLM), to simplify the issuance, verification, and management of VCs and DIDs. The implementation involves a detailed analysis of current DID and VC stacks to identify gaps where AI could effectively simplify processes. We will design and implement an interoperable, open-source framework that leverages digital public infrastructure (DPI) standards. By automating processes such as credential validation, verification explanation, and error diagnosis using AI-driven approaches, the project aims to enhance accessibility and adoption. The probable implementation path includes research, community collaboration, iterative development, and rigorous testing, culminating in a robust and easily adoptable decentralized trust ecosystem.

    Evaluate use of Unikernals for scaling controller & mediator instances in CREDEBL deployments¶

    2025-LF-Decentralized-Trust Research TAC Review

    CREDEBL is an open-source platform under the LF Decentralized Trust umbrella, specializing in Decentralized Identity (DID) and Verifiable Credentials (VC) management. To enhance the performance, scalability, and security of CREDEBL deployments, this mentorship project will explore the feasibility of using unikernels for the Credo controller and mediator instances.

    Unikernels are lightweight, specialized virtual machine images that offer advantages in terms of boot time, memory footprint, and security compared to traditional container-based deployments. By evaluating their use within CREDEBL, this project aims to determine whether unikernels can provide more efficient and scalable infrastructure for large decentralized identity deployments.

    Test coverage and improved documentation for CREDEBL Platform project¶

    2025-LF-Decentralized-Trust Coding TAC Review

    The CREDEBL Platform provides a set of scalable services to manage decentralized identity and verifiable credentials (VC) systems. It enables developers to build Self-Sovereign Identity (SSI) solutions that offer enhanced privacy and security. Designed for scalability, CREDEBL supports population-scale SSI implementations while maintaining flexibility and interoperability with multiple ledger options such as Indy and Polygon, as well as ledger-less credential issuance using methods like did:web, did:key, and did:peer.

    This mentorship project aims to enhance the platform’s test coverage by implementing end-to-end (E2E) tests as the primary objective and unit test cases as a secondary priority. The current lack of comprehensive testing exposes the platform to potential regressions, reducing reliability and maintainability. Through this effort, we aim to improve software quality, ensure API stability, and provide developers with a more robust foundation for future enhancements.

    Extensible Library of DLT Connectors for Cacti¶

    2025-LF-Decentralized-Trust Coding TAC Review

    Cacti provides modules and protocols for networks built on distributed ledgers to interoperate and run transactions spanning their boundaries, like asset transfers, asset swaps, and ledger data communication. Among these modules are DLT-specific "connectors", which provide the ability to send a limited set of ledger state queries and transaction requests to networks built on DLTs. But Cacti connectors have been developed ad hoc, because each DLT is built on a different application stack with a different SDK, and also because contributors have developed connectors without being guided by a unified vision. Also, Cacti is composed of two pre-existing systems, Cactus and Weaver, each bringing its own set of connectors (called "drivers" in Weaver) with redundant as well as complementary features.

    In this project, we will create an extensible library of pluggable connectors that can be be selected and deployed based on DLT type. We will design a common development template and gRPC-based client API for connectors by inspecting existing connector implementations. Connecters will support the vendor-neutral IETF SATP standard and trigger ledger operations through the Common Operators Module (COPM). We will also integrate Cactus and Weaver further according to the Cacti architecture by merging Cactus connectors and Weaver drivers for Fabric and Corda into canonical Cacti connectors for those two DLTs.

    Scalable and Privacy-Preserving Decentralized Identity on Layer 2 Rollups¶

    2025-LF-Decentralized-Trust Coding TAC Review

    This project aims to integrate Hyperledger Aries, Indy, and AnonCreds with Layer 2 (L2) rollups (e.g., zkSync) to create a scalable, cost-efficient, and privacy-preserving decentralized identity (DID) system. This project will offload the most expensive part of identity verification—zero-knowledge proof (ZKP) generation and verification in AnonCreds— to L2, reducing the computational and gas costs of verifiable credential (VC) issuance and authentication. By using L2 rollups for credential commitments, revocation registries, and consent management, the project ensures fast, low-cost transactions while periodically anchoring state proofs to Ethereum (L1) for auditability and trust. This system will enable decentralized authentication for dApps, and Web3 platforms without relying on centralized identity providers.

    Fabric Private Chaincode and CC-Tools for privacy-sensitive applications¶

    2025-LF-Decentralized-Trust Coding TAC Review

    Fabric Private Chaincode (FPC) leverages modern Confidential Computing technology, such as Intel SGX, to enhance the protection of chaincode and their data while executed at the endorsing peers. Client applications interact with their private chaincode via the FPC Client SDK, which is responsible to encrypt and authenticate the invocation arguments before being sent to the endorsing peers. Currently chaincodes can be developed in C++ or Golang.

    Framework for Blockchain Testing in Satellite Communication¶

    2025-LF-Decentralized-Trust Research TAC Review

    Blockchain technology, originally developed for TCP/IP-based networks, has emerged as a transformative force across various industries, attributable to its decentralized, immutable, and highly secure ledger capabilities. Nevertheless, the advent of new applications in non-TCP/IP environments, such as satellite links, mesh networks, and IoT protocols (e.g. LoRaWAN), necessitate tailored adaptations to facilitate the effective integration of blockchain. This proposal presents a testing framework aimed at assessing blockchain performance in these non-TCP/IP contexts, with a particular emphasis on its application in satellite communication.

    Web3j Libraries Full Development Lifecycle¶

    2025-LF-Decentralized-Trust Coding TAC Review

    The Web3j library is an essential tool for Java and JVM-based developers looking to integrate with the Ethereum blockchain. It provides seamless Ethereum client communication, enabling developers to interact with smart contracts and blockchain networks. However, maintaining and improving Web3j while ensuring seamless adoption of the latest Ethereum Improvement Proposals (EIPs) remains a challenge for maintainers due to the lack of time and contributors.

    This project aims to enhance the core Web3j by improving and bringing up to date its component libraries as web3j-unit, web3j-evm, web3j-openapi, focusing on reducing existing issues and integrating the missing EIPs. By improving the additional Web3j libraries, we aim to streamline development workflows, enhance testability, and create a more robust framework that simplifies Ethereum development for Java and JVM-based projects. The updates will address known gaps in the libraries, ensuring compatibility with the latest Ethereum standards while improving documentation, developer experience, and maintainability.

    Web3j: Enhancing JVM Android Support with generation of Kotlin Wrappers for Solidity Smart Contracts¶

    2025-LF-Decentralized-Trust Coding TAC Review

    The Web3j-Android library is pivotal for developers integrating Android applications with the Ethereum blockchain, enabling them to leverage Web3j's robust capabilities within a mobile environment. Following the success of the previous mentorship program in 2024, the Web3j-Android library was significantly upgraded from version 4.8.8 to version 4.12.3-android, effectively resolving challenges related to coherence, functionality, and alignment with modern Android development practices.

    This project proposes to build upon the accomplishments of the prior mentorship by further enhancing and expanding the functionalities of the Web3j-Android library. Given Kotlin's prominence as the preferred programming language for Android development, this initiative aims specifically to facilitate the generation of Kotlin-based smart contract wrappers. Achieving this objective will ensure the library fully aligns with contemporary development standards for both Android applications and Ethereum blockchain technologies. After this is achieved will continue with improving the release process and the keep up to date procedure of web3j-android with the main Web3j library.

    Empirical Analysis of Cross-Chain Protocols¶

    2025-LF-Decentralized-Trust Research TAC Review

    This mentorship program aims to conduct a comprehensive empirical study of cross-chain bridges, focusing on security, performance, scalability, and cost of protocols. Cross-chain bridges are fundamental to blockchain interoperability, enabling seamless asset transfers between distributed ledger technologies (DLTs). However, their security vulnerabilities have led to over $2 billion in losses, highlighting the urgent need for rigorous analysis. This project seeks to empirically evaluate industry-standard bridges, both within Hyperledger technologies and public blockchains, to develop performance benchmarks and security insights.

    BDLS Integration into Hyperledger Fabric¶

    2025-LF-Decentralized-Trust Coding TAC Review

    Hyperledger Fabric has historically relied on crash fault-tolerant (CFT) consensus protocols like Raft and Kafka, which do not address Byzantine failures. To enhance Fabric’s security and resilience, this project aims to integrate BDLS, a scalable Byzantine Fault Tolerant (BFT) protocol, providing robust fault tolerance in adversarial environments.

    The effort involves completing the BDLS implementation within Fabric’s ordering service, ensuring seamless compatibility with Fabric’s architecture, optimizing performance, and validating its advantages over existing solutions like SmartBFT. Key tasks include refining the protocol’s leader election, message handling, and commit phases to align with Fabric’s modular consensus framework.

    The implementation path involves extending Fabric’s ordering service API, optimizing cryptographic operations, and conducting rigorous benchmarking in distributed environments. The expected outcome is a fully integrated and production-ready BDLS consensus for Hyperledger Fabric, enabling enterprises to leverage scalable BFT consensus for mission-critical applications.

    Designing and implementing trust tasks¶

    2025-LF-Decentralized-Trust Coding TAC Review

    The Trust over IP (ToIP) project has developed a new trust architecture centered around the Trust Spanning Protocol (TSP). TSP is an interoperability layer for highly assured trust relationships between endpoints in a similar way like IP is for connectivity. Many trust tasks (or reusable patterns for trusted exchanges) can be developed in a modular fashion and with minimal development cost. In this project, prospective interns can develop 1 or 2 of such trust tasks using one of the supported languages (Rust (first preference), or Typescript, Python, Go, or C) and produces pre-specification documentation of the methods. There are a long list of tasks to choose from, for example: authentic time services, authentic location services, messaging services, TRQP over TSP, credential exchange (multiple variations) over TSP, additional DID methods. We can also factor in the candidate intern's research or career interests in selecting such trust tasks to develop.
